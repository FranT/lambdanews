Modo de entrega

El examen se entregará via ssh a un servidor git privado (mediante el comando git push origin master). Sólo el autor del examen y yo tenemos acceso a su repositorio. La fecha límite de entrega será el lunes 9 de mayo a las 00:00

Deben haber generado y entregado por correo su llave pública ssh (que se encuentra en ~/.ssh/id_rsa.pub).

Por razones de seguridad es necesario modificar el archivo ~/.ssh/config y agregar lo siguiente :

Host git.lfborjas.com 
    Port 2888

Ejecutar:

git remote add origin git@git.lfborjas.com:TU_NUMERO_DE_CUENTA.git

La ejecución de ello debería resultar en algo como:

The authenticity of host 'git.lfborjas.com (173.245.73.211)' can't be established.
RSA key fingerprint is 2f:8e:8a:a0:72:f3:ad:1d:c3:d7:05:cf:d5:9b:6e:33.
Are you sure you want to continue connecting (yes/no)?

Antes de escribir yes asegúrense que la segunda línea de esa salida sea exactamente igual a lo anterior, si no lo es, escriban no y avísenme.

Recuerden que si crean nuevos archivos dentro de la carpeta deberán ejecutar git add <ARCHIVO> para que git esté al tanto de ellos si no ejecutan git add sobre un archivo, no estará incluido cuando entreguen su examen

Cada vez que hagan un cambio significativo en el código, deberán ejecutar git commit -m "Mensaje descriptivo" (Después de haber agregado los archivos pertinentes con git add). Puesto que git permite tener este tipo de control sobre el código no se aceptarán excusas de archivos perdidos por fallas técnicas o por cambios drásticos

En cualquier momento del examen, luego de hacer git commit, siéntanse libres de ejecutar git push origin master para sincronizar su repositorio local con el repositorio de entrega. El estado de este repositorio después del último push antes de la hora de terminación del examen es el que se tomará en cuenta para revisión.

Descripción

El sitio es una copia simplificada de hackernews. La mayor parte de la funcionalidad es evidente. Parte del examen es poder identificar los recursos que se deberán manejar y cómo hacerlo. Deberán valerse de la ingeniería reversa mediante la inspección del código html generado y la funcionalidad de las páginas.

  1. Un elemento de discusión está definido por un título o url y cierto contenido opcional. Si una url es provista el vínculo de dicho elemento debe dirigir a ella. Si no es provista, se debe ir a la página de discusión del elemento.
  2. Al agregar un elemento, se verifica que la URL sea válida
  3. Existen dos vistas para los elementos de discusión: en la vista raíz los elementos se encuentran ordenados por popularidad, la cual se calcula con la siguiente fórmula: Popularidad = (puntos) / (horas_desde_creación+2)^1.8
  4. En la segunda vista, los elementos se ordenan de manera no creciente por fecha de creación.
  5. Tanto un voto como un comentario en un elemento de discusión elevan la cantidad de puntos de éste.
  6. No se pueden crear elementos de discusión sin título ni comentarios sin texto

Evaluación

  1. Funcionamiento correcto, y diseño adecuado (tanto en HTML como en la aplicación, usando REST)
  2. Respeto a principios de desarrollo, en especial el de no repetirse a sí mismo (DRY) y la separación correcta de preocupaciones (separation of concerns), de modo que el código sea modular y comprensible
